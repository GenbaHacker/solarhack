# CLAUDE.md — SolarHack 開発規律

このファイルはSolarHackリポジトリのルート規約です。全セッションで自動的に読み込まれ、
以下のルールは会話の途中で緩めることを禁止します。ユーザーが明示的に「今回は例外」と
書かない限り、常時適用してください。

背景: このプロジェクトはSAWADA社の粗利を守る積算ツールです。1円のズレが赤字案件を
生みます。「速く見える完了」より「本当に動く完了」を常に優先してください。

---

## 1. 完了宣言の禁止事項（最重要）

以下はすべて**未完了扱い**とし、報告に書いてはいけません:

- `(simulated)` / 「シミュレート」 / 「(simulated fresh reload)」 — テストを実行して
  いないことの自白です。実行しない検証は検証ではありません。
- 期待値をコードに合わせて書き換えて `PASS` にすること。期待値6が実測7で通ったら、
  期待値を7に直すのは**改竄**です。落ちた原因を掘ってください。
- 「既に正しい順序」「既に実装済み」など、確認せずに書く作文。過去に4回、
  offsetTop実測で嘘が露見しています。数字で示せない完了は完了ではありません。
- 存在しないコードの完了報告。「Fix適用」と書いたなら、そのgrep結果が空でない
  ことを自分で確認してから書いてください。
- コミット直後の「Push complete」— push後に必ず
  `git log origin/main..HEAD --oneline` を実行し、**空**であることをペーストして
  ください（俗称: push証明）。commitしただけでpushしていない事故が過去に複数回。
- **直した対象が「アプリが実際に走らせる経路」かつ「デプロイされる成果物」に届いて
  いるか、確認せず完了と書くこと。** データ・コード・アセットいずれも、修正前に
  ①アプリが実際に読む/実行する経路上のファイルか、②git追跡下で本番配信されるか
  （`.gitignore`されてないか）を両方確認する。アプリが読まない/gitが運ばないものを
  直しても効果ゼロ（TASK 138で発生: アプリ非読込かつgitignore済みのファイルを
  再生成し「840 PASS」と虚偽報告。本番へ届くのは別ファイルだった）。

## 2. self-test の必須要件

- fresh reload での実行。**必ず** `localStorage.clear()` の直後にリロード。
  ブラウザのlocalStorageは前回セッションのCOST_MASTERを保持しており、`max_dc_ac_ratio`
  が1.3等に汚染されていると全計算が狂います（過去実例）。
- 汚染検知: 各シナリオ実行の直前に `COST_MASTER.max_dc_ac_ratio` の値をログ出力。
  期待値は 1.6。1.3や他の値なら汚染。
- 数値検証と目視検証は別項目です。JSの `textContent` 読取は `display:none` の
  要素も貫通します。「値は取れるが画面に映らない」事故が過去に発生（TASK 123-FIX2、
  126で二度）。UIを含むタスクは**必ずスクリーンショット**を残し、視認できる
  画面での動作を確認してください。
- テストシナリオ: 既存の `V2_loadTestScenario('multi'|'mixed'|'lowv'|'boundary'|'hicurrent')`
  を使用。実装したタスクに関係なくても、multi と mixed の回帰は毎回実行:
  - multi: `pcsUnits === 6` AND `V2_EST.cost === 32290473.2`（±0.5%以内）
  - mixed: `panelCost === 419734` AND `V2_EST.cost === 4280762.2`（低圧時）
  この2件が崩れた瞬間、実装は失敗です。
  ※基準は2026-07-17に現行原価へ再設定（旧: multi 40-45M / mixed panelCost386974・原価3555999.2。
    パネル単価更新・監視/その他機器の原価組込みによる正当な変化を反映）。原価が下振れした場合は
    粗利過大評価→赤字見逃しに直結するため、基準割れは必ず回帰調査すること。
  - 粗利率(grossMargin): markup_equipment=0.20 / markup_work=0.25 で multi/mixed とも約36%。
    目標レンジ30〜40%。三重錠の下限20%を割る変更は禁止。マークアップ既定を変えたら
    COST_MASTER_SCHEMA_VERSION を+1し、施主ブラウザのlocalStorage影を自動移行させること
    （V2_loadMaster の価格スキーマ移行処理。手動console削除は不要になった）。
- **発電・シミュ精度のPASSは、偽装不能な外部実測と照合すること。** 内部整合や
  未検証ファイルへのスポットチェックはPASS根拠になりません。このツールの価値は
  「数字が実績と合うか」です。日射・発電量の検証は FusionSolar等の実測との
  月別Pearson相関 r>0.85 かつ 年間±15% を最低ラインとします（邑南町418.56kWp/
  2025実績で校正済み）。

## 3. 1タスク1コミットの原則

- 1コミットに複数タスクを混ぜないこと。差し戻し時に切り分け不能になります。
- コミット前に必ず parse check（node --check や構文検証）。
- コミットメッセージ形式: `feat/fix/chore: TASK <番号> — <要約>`
- タスク仕様書に「A/B/C/D」など分割単位がある場合は、分割単位ごとにコミット。

## 4. 差分編集の原則（ユーザーの明示ルール）

- HTMLファイルの修正は`str_replace`（差分編集）で行う。全体書き直しは禁止。
  トークン消費が桁違いに増え、意図しない他箇所の変更を混入させる原因になります。
- 新規HTML生成が必要な場合は、まず骨格のみを提示してユーザー承認を取ってから
  詳細を追加。
- 1レスポンスで出力するコードは300行以内を目安。

## 5. 禁止事項

- 存在しないファイルの自作。「denkihack.htmlが無い」と分かったら、STOPして
  ユーザーに報告してください。**自分で作ってはいけません。** ユーザーが手動で
  デプロイするファイルは、あなたの管轄外です。
- 「your part」等、self-testをユーザーに丸投げする文言。self-testは実装者
  （あなた）の仕事です。ユーザーは施主検査（本番での最終確認）を担当します。
- 「blockingするalert」の使用。過去に45秒のCDPフリーズを引き起こし、
  検証を止めました（TASK 123系）。確認表示は`<span>`のtextContent更新で行う。

## 6. 状況変化の検知（環境系の落とし穴）

- Actions が実行中（デプロイ中）に本番URLをprobeすると、旧版が返ってきて偽の
  失敗が発生します。デプロイ完了を確認してから検証してください（俗称:
  ぐるぐるタイム / 名札のない⏳禁止）。
- ローカルクローンが古い可能性: gate checkで「ファイル無し」と出た場合、
  `git pull --rebase origin main` を試す。ユーザーがGitHub Web UIから直接
  コミットしている可能性があります（実例あり）。
- **校正定数はlocalStorageに影を持つ**: `sim_base_coefficient`(K)や
  `max_dc_ac_ratio`等をCOST_MASTERのハードコード既定で変更しても、`V2_loadMaster`が
  保存済み`localStorage['COST_MASTER']`で上書きするため、既存ユーザー（施主）の
  ブラウザでは旧値が復活し変更が無効になります。定数変更時は、該当キーをlocalStorageから
  削除（キー単位パッチ、他の保存値は温存）する手順を必ず併記・実行すること
  （TASK K-calibでsim_base=0.92の影を除去した実例）。

## 7. 出力形式（ユーザーの明示ルール）

- 回答は日本語。
- タスク仕様書は英語骨格＋日本語UI/業務要件のハイブリッド形式（過去のTASKで
  実証済み）。この形式が最も指示ミスを起こしにくいです。
- コード提示時は変更箇所と理由を明確に。エラー時は原因と解決策を両方。

## 用語集（GENBA HACKER内での共通語）

- **施主検査**: ユーザーが本番URLで行う最終確認。実装者の自己検査（self-test）
  とは別物。self-testが緑でも、施主検査で赤なら未完了です。
- **ぐるぐるタイム**: GitHub Actionsのデプロイ待ち時間（実測1〜3分）。
  この間の本番probeは禁止（旧版が返る）。
- **push証明**: `git log origin/main..HEAD --oneline` の空出力。
- **汚れた机**: localStorage / sessionStorage に前回セッションの状態が
  残っていて、fresh reloadでも汚染が消えない現象。`.clear()`で拭く。
- **名札のない⏳**: 何のデプロイを待っているか特定できないぐるぐる。
  Actionsのコミット名で照合してから待つこと。

## 現在の設計不変条件（変更禁止）

以下はSolarHack v2の骨格。壊した瞬間、赤字案件検知機能が死にます:

1. **単一台帳**: 原価は`V2_EST.cost` ただ一つ。DOMからの逆算やEQUIP_PRICESからの
   別計算は禁止。過去に二重計上バグで排除済み。
2. **三重錠**: 未確認リスク・粗利率・日射チェックのゲート。売価表示はゲートを
   通ってから。
3. **モード分離**: クイックモード（面ゼロ・totalPanels直接入力）と設計モード
   （v2RoofFaces配列あり）の分離。`V2_syncTotals`は`if (hasDesign)`ガード必須
   （TASK 122-F）。
4. **概算/詳細二タブ構造**: 概算タブは営業がその場で内諾を取る道具。詳細タブは
   現場調査後の実査。差分ブロックが手綱。この構造を壊す変更は禁止。
5. **DenkiHackとの双方向ブリッジ**: `SH2DH_HANDOFF` / `DH2SH_RESULT` のキー名と
   ペイロード構造は固定。denkihack.html側の実装と一対一で対応（TASK 127-FINAL）。
